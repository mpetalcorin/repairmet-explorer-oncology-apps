# Datasheet for RepairMet Explorer Inputs and Outputs

## Overview

This datasheet describes the data structures expected and generated by RepairMet Explorer. The platform does not currently depend on a single proprietary dataset. Instead, it operates on a standardized feature representation that can be derived from multiple upstream evidence streams.

## What counts as “data” in this project

There are three main data layers:

1. input feature values
2. optional evidence-weight values
3. generated output records

In cohort mode, each row corresponds to one sample and each column corresponds to one normalized or human-interpretable feature.

## Input data requirements

### Required sample fields

Each sample must include:

- sample_name
- hrd
- fork
- repStress
- atr
- glycolysis
- oxphos
- glutamine
- ros
- redox
- lactate

### Optional evidence-weight fields

For API scoring, evidence weights may also be supplied:

- pathway
- assay
- omics
- expert

### Expected scale

All feature values are expected on a 0 to 100 scale before internal normalization.

Example:

- 0 = absent, minimal, or very low signal
- 100 = strong, dominant, or highly supported signal

The app internally converts these values to the interval from 0 to 1.

## Biological meaning of each feature

### sample_name
Human-readable identifier for the sample.

### hrd
Homologous recombination deficiency signal.

### fork
Replication-fork instability signal.

### repStress
Replication-stress burden signal.

### atr
ATR-CHK1 checkpoint signaling signal.

### glycolysis
Degree of glycolytic reliance.

### oxphos
Degree of oxidative phosphorylation competence.

### glutamine
Degree of glutamine dependence.

### ros
Reactive oxygen species pressure.

### redox
Redox buffering or antioxidant defense signal.

### lactate
Lactate handling or acid-waste management signal.

## Acceptable upstream sources

These feature values may be derived from:

- expert-curated biological interpretation
- functional assays
- pathway activity inference
- transcriptomic summaries
- proteomic summaries
- metabolomic summaries
- integrated multi-omics processing
- simulated tumour-state scenarios for software testing

## Data formatting requirements

### CSV format

Expected CSV header:

```text
sample_name,hrd,fork,repStress,atr,glycolysis,oxphos,glutamine,ros,redox,lactate
```
## Data validation rules

The app expects:
	•	non-empty sample_name
	•	all required feature columns present
	•	numeric values for all features
	•	no missing feature entries within a row
	•	feature values intended for the 0 to 100 range

Invalid rows may produce upload failure or misleading rankings.

## Output data schema

For each scored sample, the platform returns:
	•	sample_name
	•	normalized_features
	•	composite_state
	•	ranked_hypotheses

### normalized_features

Dictionary of input features converted to 0 to 1 scale.

### composite_state

Contains:
	•	repairDefect
	•	replicationBurden
	•	glycolyticAcidLoad
	•	mitoPlasticity
	•	redoxFragility

### ranked_hypotheses

List of ranked vulnerability objects. Each object contains:
	•	id
	•	title
	•	mechanism
	•	next_step
	•	score
	•	base_score
	•	evidence_multiplier
	•	explain

### explain

Top contributing feature list for that hypothesis. Each contributor contains:
	•	label
	•	score

### Data provenance considerations

Because the platform accepts interpreted feature scores, provenance should be tracked outside the app whenever possible. For rigorous research use, users should record:
	•	the source assay or dataset
	•	preprocessing steps
	•	normalization method
	•	scoring rationale
	•	analyst or reviewer identity
	•	date and version of the exported result

### Data quality considerations

The app works best when the feature layer is:
	•	biologically interpretable
	•	consistently scaled
	•	quality controlled
	•	comparable across samples
	•	derived from documented evidence streams

The app may work poorly when:
	•	features are loosely estimated without rationale
	•	cohorts have major batch effects
	•	different samples use incompatible scoring schemes
	•	the biology is forced into the wrong feature space

## Data limitations

This project does not yet ship with:
	•	a single benchmark dataset
	•	a clinical outcomes dataset
	•	tumour-type-specific reference cohorts
	•	probabilistically calibrated training labels

The current version is designed around structured mechanistic interpretation, not supervised clinical prediction.

## Recommended documentation for real projects

For a publication-quality or team-shared cohort, record at minimum:
	•	project name
	•	sample source
	•	tumour type or model type
	•	feature derivation method
	•	evidence-weight rationale
	•	software version
	•	export date
	•	downstream decisions or assay follow-up

## Privacy and sensitivity

If real patient-linked samples are used upstream, users are responsible for ensuring that all exported files and uploaded cohort tables are handled according to their local ethical, privacy, and data-governance requirements.

## Maintenance notes

As the hypothesis library expands, the expected feature space may grow. This datasheet should be versioned alongside the software so that users know which inputs and outputs correspond to which release.
